# V1 UI Restructure Plan

## Reason

The first V1 implementation proved the workflow logic:

- Records can be added.
- Records can be selected.
- Status can be updated.
- Next action can be updated.
- Tasks can be added.
- Handoff notes can be added.
- Activity history is logged.
- Demo data persists locally.

The current page layout stacks too many controls vertically. It works for testing, but it can become confusing for real users and will not scale well when there are many records.

## UI Direction

Move from a long stacked page to an app-style workspace.

Recommended layout:

1. Product header.
2. Summary dashboard.
3. Main workspace.
4. Left side: compact record list.
5. Right side: selected record detail.
6. Detail area uses tabs.

## Detail Tabs

Use these tabs:

- Overview.
- Next Action.
- Work Items.
- Handoff Notes.
- Activity.

## Copy Decisions

Use user-facing language.

Keep:

- Next Action.

Rename:

- Workflow Tasks -> Work Items.

Definitions:

- Next Action: the one action that should happen next for this lead or client.
- Work Items: supporting tasks for follow-up, onboarding, delivery, approvals, payments, or handoff.

## Scaling Requirements

The record list should prepare for:

- Search.
- Stage filter.
- Risk filter.
- Owner filter.
- Priority filter.
- Sort by follow-up date.

These filters do not all need to be built immediately, but the layout should support them.

## Add Record Pattern

The add-record form should not remain as a large always-visible section.

Recommended V1 pattern:

- Add Lead Or Client button.
- Form appears in a modal or collapsible panel.
- After submit, the new record is selected.

## V1 Restructure Build Order

1. Rename Workflow Tasks UI copy to Work Items.
2. Create tab state for selected record detail.
3. Split detail panel into tab sections.
4. Move Next Action form into the Next Action tab.
5. Move Work Item list/form into the Work Items tab.
6. Move Handoff Note list/form into the Handoff Notes tab.
7. Move Activity History into the Activity tab.
8. Move Add Record form behind a button.
9. Add compact list structure that can later support search and filters.
10. Retest all existing V1 workflow behavior.

## Definition Of Done

The UI restructure is complete when:

- The app no longer feels like one long stacked form.
- Selected record details are organized into clear tabs.
- Next Action and Work Items are visually distinct.
- Add Record is not always visible.
- Existing workflow behavior still works.
- Local persistence still works.
- Mobile layout remains usable.
- No horizontal scroll.

# V2 Backend, Auth, And Storage Plan

## Goal

Move the Client Operations & Revenue Workflow System from local browser storage to real account-based storage for private beta users.

V2 should support real users safely, but still avoid overbuilding.

## Recommended V2 Stack

- Frontend: Next.js + TypeScript.
- Auth: Supabase Auth.
- Database: Supabase Postgres.
- Security: Row Level Security.
- Server logic: Next.js API routes/server actions first.
- Future backend: Python + FastAPI for AI, reminders, workers, webhooks, and integrations.

## Why Supabase First

Supabase is a good V2 choice because it provides:

- user authentication.
- Postgres database.
- row-level security.
- hosted backend quickly.
- SQL access for clear data modeling.
- easier private beta setup.

FastAPI should come later when the product needs:

- AI workflows.
- scheduled reminders.
- background jobs.
- webhook endpoints.
- integration orchestration.
- audit/retry logic.

## V2 Product Boundary

V2 should add:

- sign up / sign in.
- workspace records saved to database.
- user-owned records.
- activity logs saved to database.
- handoff notes saved to database.
- work items saved to database.
- safe loading/error states.
- ability to reset sample workspace separately from real data.

V2 should not add yet:

- billing.
- multi-user teams.
- external integrations.
- autonomous AI actions.
- payment processing.
- full CRM complexity.

## Data Model

### profiles

Stores user profile data.

Fields:

- id uuid primary key references auth.users(id)
- email text
- full_name text
- created_at timestamp
- updated_at timestamp

### workspaces

Stores a user workspace.

Fields:

- id uuid primary key
- owner_id uuid references auth.users(id)
- name text
- created_at timestamp
- updated_at timestamp

### client_workflow_records

Stores leads and clients.

### workflow_tasks

- action_type text
- note text
- created_at timestamp

## Security Model

Use Row Level Security.

Every table with product data should include `workspace_id`.

Users should only access rows where they own the workspace.

V2 private beta can use a simple model:

- one user owns one workspace.
- no team members yet.
- no role-based permissions yet.

Later V5 can add:

- team members.
- roles.
- assigned owners linked to users.
- workspace invitations.

## Auth Flow

V2 should support:

- sign up.
- sign in.
- sign out.
- protected workspace.
- loading state while auth is checked.

User flow:

1. User visits product.
2. User can view sample workspace without signing in.
3. User signs up to save real workspace data.
4. App creates profile and workspace.
5. User creates records, work items, notes, and updates.
6. Data is saved to Supabase.

## Sample Workspace vs Real Workspace

Keep these separate.

Sample workspace:

- safe demo data.
- no login required.
- browser/local only or public static seed.
- resettable.

Real workspace:

- requires auth.
- saves to Supabase.
- should warn users not to store unnecessary sensitive data.
- belongs to the logged-in user.

## API Strategy

Start simple.

Option A:
Use Supabase client from the browser with RLS.

Option B:
Use Next.js API routes/server actions for mutations.

Recommended:
Use Supabase client with RLS for early V2 CRUD, but use API routes later for sensitive workflows, AI calls, reminders, and admin actions.
